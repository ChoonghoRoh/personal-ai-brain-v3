## 현재 기능 사용 경험

- **청크 생성**: 고객사 요구사항 문서 매일 10~20건, CI/CD·GitHub Markdown 자동 수집 시도.  
  **느낀 점**: "API 문서 부족. 파일 업로드·분할 한 번에 처리 API 없어 3번 호출 필요."
- **청크 승인**: 조건부 자동 승인(신뢰도>0.8) 희망.  
  **느낀 점**: "승인 API가 1건씩만. 100개 승인에 100번 호출 → 느리다. Bulk API 필요."
- **청크 관리**: Slack 봇에서 검색·고객 답변, 라벨 API 자동화.  
  **느낀 점**: "Rate Limit 없어 걱정. 1초 10번 검색 시 서버 다운 가능. Rate Limiting·Caching 필요."

---

## 사용 불편점

| 불편점 | 심각도 | 영향 |
|--------|--------|------|
| API 문서 부족 | 상 | 엔드포인트 찾기 어려움 |
| Bulk API 없음 | 상 | 100개 승인 100번 호출 → 느림 |
| Rate Limiting 부재 | 중 | DDoS 취약 |
| Webhook 없음 | 중 | 청크 생성 완료 시 자동 알림 불가 |

---

## 변경 요구사항

1. API 우선 설계 (OpenAPI 3.0, Swagger UI)  
2. Bulk API (bulk-create, bulk-approve)  
3. Webhook (청크 생성 완료 → Slack/Discord, 승인 완료 → GitHub Issue)  
4. Rate Limiting·Caching (IP당 10req/s, Redis 캐싱)

---

## 업그레이드 시나리오 제안

- **옵션 A**: GitHub Actions → bulk-create → Webhook(Slack)·자동 승인 → 고객 Slack 봇(검색·캐시·Rate Limit)  
- **옵션 B**: Python/TypeScript SDK, CLI (`pab-cli upload file.md`)

## 기대 효과 (스타트업 개발 속도 관점)

| 개선 항목 | 현재 문제 | 개선 후 |
|-----------|-----------|---------|
| API 문서 | 구글링·시행착오 4시간 | Swagger UI → 30분 |
