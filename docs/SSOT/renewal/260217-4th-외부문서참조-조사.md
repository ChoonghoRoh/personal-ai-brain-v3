## 2. docs/phases/ 참조 (4th 외부)

Phase 실행 시 읽는 파일 경로. 4th 워크플로우 정의상 **프로젝트 표준 위치**라서 4th 안으로 옮기기 어렵고, 그대로 두는 것이 자연스러움.

| 4th 내 파일 | 참조 내용 | 외부 경로 |
|-------------|-----------|-----------|
| **0-entrypoint.md** | ENTRY-1 단일 진입점 | `docs/phases/phase-X-Y/phase-X-Y-status.md` |
| **0-entrypoint.md** | Phase Chain 사용 | `docs/phases/phase-chain-{name}.md` |
| **3-workflow.md** | ENTRY-1, §2.1 파일 경로, §8.2 Chain 정의 | `docs/phases/phase-X-Y/phase-X-Y-status.md`, `docs/phases/phase-chain-{name}.md` |

- **역할**: Phase별 상태 파일·Chain 정의 파일의 **저장 위치** 명시.
- **권장**: 유지. phases는 프로젝트 전역 구조이므로 4th와 독립된 경로가 맞음.

---

## 3. docs/rules/ 참조 (4th 외부)

4th **1-project.md** §8 참조 문서 테이블에서만 사용.

| 4th 내 파일 | 참조 내용 | 외부 경로 |
|-------------|-----------|-----------|
| **1-project.md** | 검증 템플릿 | `docs/rules/templates/verification-report-template.md` |
| **1-project.md** | Task 생성 규칙 | `docs/rules/ai/references/ai-rule-task-creation.md` |
| **1-project.md** | Task 검사 규칙 | `docs/rules/ai/references/ai-rule-task-inspection.md` |
| **1-project.md** | Plan·Todo 생성 규칙 | `docs/rules/ai/references/ai-rule-phase-plan-todo-generation.md` |

- **역할**: Task/Plan 생성·검증 시 참고하는 **프로젝트 공통 규칙** 문서.
- **선택**: 4th 완전 단독화를 원하면 이 4개 규칙을 4th 내부(예: 4th/rules/ 또는 GUIDES 보강)로 요약·복사할 수 있음. 현재는 프로젝트 공통 규칙을 가리키는 링크로만 사용.

## 4. docs/rules/role/ (설명·출처만, 링크 아님)

4th **내부** Charter 링크는 모두 PERSONA/*.md로 바뀌어 있음.  
아래는 “원본이 docs/rules/role/ 이었다”는 **설명·출처** 문구만 있음.

| 4th 내 파일 | 내용 |
|-------------|------|
| **PERSONA/README.md** | “`docs/rules/role/` Charter 내용을 4th SSOT 내로 복사·통합” 등 설명 |
| **PERSONA/LEADER.md, BACKEND.md, FRONTEND.md, QA.md** | 헤더에 “**출처**: `docs/rules/role/...` → 4th PERSONA로 통합” |
| **VERSION.md** | “docs/rules/role → 4th 통합”, “PERSONA 통합” 등 변경 이력 설명 |

- **역할**: 출처 표기·이력 설명. **문서 링크가 아님**.
- **권장**: 유지해도 됨. 4th 단독 사용 시 실제 로딩은 PERSONA/*.md만 하면 됨.

## 5. 프로젝트 디렉터리 경로 (문서 아님)

4th 문서 안에 나오는 **코드·테스트·리소스** 경로. “상위 폴더의 문서”가 아니라 **작업 대상 디렉터리**임.

| 경로 | 나오는 4th 문서 예 |
|------|---------------------|
| **backend/** | 0-entrypoint, 1-project, 2-architecture, ROLES/backend-dev, GUIDES/backend-work-guide |
| **web/** | 0-entrypoint, 1-project, 2-architecture, ROLES/frontend-dev, GUIDES/frontend-work-guide |
| **tests/** | 1-project, ROLES/backend-dev, ROLES/tester, GUIDES/backend-work-guide, GUIDES/tester-work-guide |
| **e2e/** | 1-project, ROLES/tester, ROLES/frontend-dev, GUIDES/tester-work-guide |

- **역할**: 팀원 편집 범위·테스트 명령·디렉터리 구조 설명.
- **권장**: 유지. 리포지터리 루트 기준 경로이므로 4th와 무관하게 프로젝트 구조를 가리키는 것이 맞음.

---

## 6. 정리

- **4th 밖의 “문서”를 참조하는 곳**  
  - **docs/phases/** : 0-entrypoint, 3-workflow (Phase status·Phase Chain 경로).  
  - **docs/rules/** : 1-project 참조 문서 테이블 4건 (templates, ai/references).

- **4th 완전 단독 사용**을 위해 꼭 손댈 필요는 없음.  
  - phases, rules 경로는 **프로젝트 공통 위치**를 가리키는 것이고, 4th는 “그 위치를 쓰라”고만 정의함.
- **완전 단독화**를 원하면:  
  - `docs/rules/` 쪽 4개 규칙만 4th 내부에 요약본을 두고, 1-project 참조를 그 요약본으로 바꾸는 방식을 검토할 수 있음.
