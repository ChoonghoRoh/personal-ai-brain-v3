## 3. 관점별 요약

### 3.1 꼼꼼한 기획자 관점 (Planner)

| 항목                | 요약                                                                                                                                                       |
| ------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **요구사항·플로우** | Phase 9-3 체크리스트와 대응하는 화면·진입 경로가 일치하며, 메뉴 진입 → 화면 동작 → 결과 확인 흐름이 자연스럽게 구성됨.                                     |
| **안내·문서화**     | 라벨·버튼이 대체로 명확함. 예외(검색 무결과, API 오류, 존재하지 않는 청크 등) 시 사용자 안내 문구를 요구사항/기획서에 명시하면 일관된 UX 기획에 도움이 됨. |
| **주요 제안**       | 예외/에러 시나리오별 사용자 안내 문구를 요구사항 문서에 추가. LLM 연동 시나리오(W3.2, W3.4, W4.2~W4.7)는 Ollama 등 환경 준비 후 재검증 권장.               |

### 3.2 개발자 관점 (Developer)

| 항목               | 요약                                                                                                                                                                                                              |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **버그**           | E2E 19항목 통과. 현재 스펙 범위 내 치명적 버그 미발견.                                                                                                                                                            |
| **콘솔/API**       | E2E 실행 시 JavaScript 에러·실패한 네트워크 요청 없음. Reasoning 실행 시 로딩 UI·버튼 비활성화 적용됨.                                                                                                            |
| **추가 검증 권장** | (1) Ask/Reasoning LLM 연동: API 타임아웃·5xx 시 클라이언트 에러 메시지·재시도 (2) chunk_id 없음/잘못된 id: API·UI 응답 일관성(404 vs 200+빈 데이터) (3) 청크 생성 1단계: 스토리지 목록 실패 시 에러 메시지·재시도 |

### 3.3 UI·UX 디자이너 관점 (Designer)

| 항목                   | 요약                                                                                                                                                            |
| ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **버튼/링크·레이아웃** | 주요 CTA가 버튼/링크로 구분되어 있으며, 대시보드·검색·Ask·Reasoning·지식·관리 페이지가 container·panel 구조로 일관됨.                                           |
| **접근성**             | 포커스 표시, 키보드 조작, 라벨/placeholder는 코드 상 존재. 탭·필터 등에 aria·role 등 추가 점검 권장.                                                            |
| **개선 제안**          | 검색/결과 없음·빈 청크·미승인 목록 빈 경우 등 빈 상태 메시지 가독성·위치 확인. 로딩/비활성화 상태 시각적 명확성, 단계별 진행 상태(청크 생성 1·2·3) 명확화 권장. |

## 4. 통합 액션 아이템 (개선 권장 사항)

다음 항목은 3관점 보고서에서 공통·우선 반영을 권장하는 사항이다.

| 우선순위 | 구분          | 액션                                                                                                                                | 근거 관점          |
| -------- | ------------- | ----------------------------------------------------------------------------------------------------------------------------------- | ------------------ |
| P1       | 요구사항·안내 | 예외/에러 시나리오별 사용자 안내 문구를 요구사항 문서에 명시 (검색 무결과, API 오류, 존재하지 않는 chunk_id, 스토리지 목록 실패 등) | Planner            |
| P1       | 개발·API      | chunk_id 없음/잘못된 id에 대한 API 응답 정책 정리 (404 vs 200+빈 데이터) 및 UI 메시지 일관화                                        | Developer, Planner |
| P1       | 개발·UI       | Ask/Reasoning LLM 연동 구간: API 타임아웃·5xx 시 클라이언트 에러 메시지·재시도 노출 확인                                            | Developer, Planner |
| P2       | UI/UX         | 빈 상태 메시지(검색 결과 없음, 미승인 청크 없음, 청크 없음 등) 가독성·위치·색상 확인                                                | Designer, Planner  |
| P2       | UI/UX         | 로딩 중("고민 중... N초")·버튼 비활성화 상태 시각적 명확성 확인                                                                     | Designer           |
| P2       | 개발          | 청크 생성 1단계: 스토리지 목록 로드 실패 시 에러 메시지·재시도 노출 확인                                                            | Developer          |
| P3       | 접근성        | 탭·필터 등에 aria-selected, role 등 접근성 속성 적용 여부 페이지별 점검                                                             | Designer           |
| P3       | 검증          | W3.2, W3.4, W4.2~W4.7(LLM 연동), W11.2/W11.3, W12.2/W12.3/W12.5(데이터·조작) 실제 환경·데이터로 재검증                              | Planner, Developer |

## 5. 결론

- **전체 평가**: Phase 9-3 웹 체크리스트에 대응하는 E2E 19항목이 모두 통과하였고, 기획(플로우·요구사항)·개발(버그·API)·UI·UX(시인성·일관성) 관점에서도 현재 범위 내에서는 큰 결함 없이 동작하는 것으로 정리되었다.
- **한계**: LLM(Ollama) 연동 구간, 실제 데이터 기반 조작(청크 승인·라벨 적용·청크 생성 2·3단계 등)은 별도 환경·데이터 준비 후 수동 또는 E2E 확장 검증이 필요하다.
- **다음 단계**: 위 P1·P2 액션 아이템을 요구사항·개발·디자인 반영 후, 필요 시 동일 체크리스트와 관점별 보고 형식으로 재검증하는 것을 권장한다.
